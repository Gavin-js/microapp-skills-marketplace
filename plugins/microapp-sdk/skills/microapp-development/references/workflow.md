## 通用规则

- 先确认需求，再写代码。
- 先确定持久化策略，再设计界面状态。
- 先定义移动端和 PC 端职责，再决定页面结构。
- 能从 URL、业务参数或 SDK 读取的信息，不要重复向用户索取。

## 通用提问清单

每次都至少确认以下字段：

| 字段 | 说明 |
| --- | --- |
| `appType` | `lottery-signin` / `signin` / `lottery` / `survey` / `other` |
| `platform` | `mobile` / `web` / `both` |
| `designStyle` | `cyber` / `ink` / `crystal` / `celebration` |
| `storageMode` | 自建表 / CRM 对象 / 本地临时演示 |

补充规则：

- `lottery-signin` 和 `signin` 默认推荐 `both`。
- `survey` 和 `other` 默认先按单端方案收敛，除非用户明确要求大屏或联动。
- 未给风格时，默认 `cyber`，但仍要说明可以切换。

## 按应用类型确认需求

### `lottery-signin`

默认能力：

- 移动端签到
- PC 端签到名单展示
- 互动消息滚动
- 签到二维码
- 多轮抽奖
- 奖项管理

必须确认：

- 奖项设置：每个奖项的名称、数量、奖品说明
- 存储方式：自建表、CRM 对象还是临时演示
- 实现方式：自建数据流还是系统参会人员对接

推荐默认值：

- 使用自建表存储签到和中奖结果
- 每人只允许中奖一次
- 移动端签到后立即向 PC 端广播刷新事件

### `signin`

默认能力：

- 用户签到
- 签到名单实时刷新
- 二维码展示
- 互动消息滚动

可选增强：

- CTA 引导
- 额外信息收集
- 头像飞入或签到球等大屏动效

必须确认：

- 是否需要额外表单字段
- 是否接 CTA
- 是否使用系统会议签到接口

### `lottery`

默认能力：

- 奖项管理
- 多轮抽奖
- 中奖名单
- 中奖偏好设置

必须确认：

- 是否接 CTA
- 奖项设置
- 抽奖池来源：系统参会人员、用户报名、导入名单或临时数据
- 存储方式

默认规则：

- 抽奖前先持久化奖项配置
- 抽奖后先持久化中奖记录，再广播结果

### `survey`

必须确认：

- 题目列表
- 题型和选项
- 是否限制单次提交
- 提交后是否允许修改
- 存储方式

默认规则：

- 如果用户未说明，按每人一次提交处理。
- 如果没有特殊要求，优先使用自建表存储问卷答案。

### `other`

必须确认：

- 目标功能
- 目标端形态
- 需要保存的数据
- 是否依赖微应用 SDK 能力

## 方案收敛原则

### 端职责

双端场景优先这样拆分：

| 端 | 职责 |
| --- | --- |
| `mobile` | 参与、签到、提交、发送消息、接收结果通知 |
| `web` | 大屏展示、名单刷新、主持人操作、二维码展示、开奖或统计 |

### 数据职责

- 用 `db.*` 保存结构化业务数据，例如签到记录、中奖记录、问卷答案。
- 用 `storeAppData()` / `getAppData()` 保存轻量配置，例如奖项配置、中奖偏好、当前主题。
- 只有在用户明确要求 CRM 对接时，才用 `object.*` 读写业务对象。

### 参数职责

这些值优先从运行环境读取，而不是先问用户：

- `ea`
- `appId`
- `marketingEventId`
- `ctaId`

只有以下情况才追问：

- 当前方案强依赖该值，但页面运行环境不保证提供
- 用户要求你生成固定链接、固定二维码或硬编码示例
- CRM 对象方案需要明确的 `apiName` 和字段定义

## 生成前检查

开始写代码前，至少确认：

- 已经知道 `appType`
- 已经知道端形态
- 已经决定持久化方式
- 已经列出需要的表或配置键
- 已经明确哪些交互需要双端同步
